date

Functions that manipulate or generate dates.
Most date functions support modifying, setting, offsetting, or rounding a date with respect to one or more time periods.
The supported time periods
PeriodFull NamesShort NameRangeNotes
Yearsyear, yearsy-999,999,999 to 999,999,999
Monthsmonth, monthsM1 to 12
Weeksweek, weeksw0 to 54[1]
Daysday, daysd1 to 28, 29, 30, or 31
Hourshour, hoursh0 to 23
Minutesminute, minutesm0 to 59
Secondssecond, secondss0 to 59
Millimilli, millisI0 to 999[2]
Micromicro, microsU0 to 999[2]
Nanonano, nanosN0 to 999[2]
Milli of secondmilliOfSecond, millisOfSecondi0 to 999[3]
Micro of secondmicroOfSecond, microsOfSecondu0 to 999,999[3]
Nano of secondnanoOfSecond, nanosOfSecondn0 to 999,999,999[3]

Notes:

  1. The first day of a week and the first week of a year are set in the EEL context. By default, the first day of the week is Monday, and there are at least 4 days in the first week, following ISO 8601.
  2. Milli, Micro, and Nano are independent. For example, setting microseconds does not affect milliseconds or nanoseconds.
  3. Milli of Second, Micro of Second, and Nano of Second update all fractional parts of a second. For example, setting microseconds to 1,234 sets millis to 1, micros to 234, and nanos to 0. For adding, subtracting, or snapping, these periods act like their independent counterparts.

Period names are case-sensitive. Time zones can be:

  • Fixed offsets – a resolved offset from UTC such as +5
  • Geographical regions – a region following specific rules to determine offsets, e.g., Europe/Paris

Time periods can be snapped to the start of a period with a modifier @<period>. Snap modifiers always round down.

Examples of valid modifiers:

  • 7d : 7 days in the future
  • -7d : 7 days in the past
  • 7days : 7 days in the future using the full name
  • @d : the start of the current day

Offsets are applied in the order passed. Examples:

  • date.utc() – current UTC date-time with nanosecond accuracy
  • date.utc("@s") – current UTC date-time with second accuracy
  • date.utc("1day") – current UTC date-time plus 1 day
  • date.utc("@d", "+12h") – current date at midday UTC
  • date.utc("@d", "-1_500n") – 1500 nanoseconds before start of UTC date
  • date.utc("1d", "@d") – midnight tomorrow UTC
  • date.utc("48h", "@d") – midnight the day after tomorrow UTC
  • date.start("Europe/Paris") – instant the EEL context was created in Paris
  • date.local("+2months") – current local date-time plus 2 months
  • date.at("-5", "-1w") – current date-time in UTC-5 minus 1 week
  • date.set(${value}, "12h", "0m", "0s", "0n") – given date with fields set to midday
  • date.plus(${value}, "15minutes") – given date plus 15 minutes
  • date.minus(${value}, "-15minutes") – given date minus 15 minutes
  • format.at("Europe/Paris", "d/M/yyyy HH:mm") – current Paris date and time in given format

date.at(zone, offsets...)

Returns the current date/time in the specified zone with optional offsets applied. The returned value is evaluated at the time of invocation.

Called as:

date.at(zone)
date.at(zone, offsets...)

Parameters:

zone
Type : Text
Description : The time zone
offsets...

Default Value : No offsets
Description : The offsets to apply to the current timestamp (for example, years, months, days, or hours), applied sequentially

Return type:

Date

Returns:

The current date/time in the specified zone with the offsets applied

DateTimeException If the zone is not a valid time zone ID

date.parse(pattern, date, offsets...)

Parses text representing a date/time using the specified DateTimeFormatter pattern.

The pattern must follow the syntax defined by DateTimeFormatter. The date text must conform exactly to this pattern.

Any temporal fields not defined by the pattern are assigned default values as follows:

  • Era defaults to AD (CE)
  • Time zone defaults to UTC if not specified by the pattern
  • Year defaults to the current year at the time of parsing
  • All other fields default to their lowest valid values (e.g., month = 1, day = 1, hour = 0)
Fields present in the pattern must be present in the date text and are parsed strictly.

Called as:

date.parse(pattern, date)
date.parse(pattern, date, offsets...)

Parameters:

pattern
Type : Text
Description : The DateTimeFormatter pattern describing the expected structure of the date text
date
Type : Text
Description : The date/time text to parse, which must exactly match the specified pattern
offsets...
Type : Text
Default Value : No offsets
Description : Optional offsets to apply to the timestamp, applied sequentially

Return type:

Date

Returns:

The parsed date/time value with any missing fields defaulted as described above

Throws:

DateTimeException If the pattern is invalid or the date doesn't match the pattern

See:

DateTimeFormatter documentation

Since:

date.start(zone, offsets...)

Returns the date/time at which the given context was created, adjusted to the specified time zone.

The returned value represents a fixed timestamp associated with the creation of the context. It does not change across repeated evaluations using the same context.

The timestamp is converted to the specified zone. Any supplied offsets are then applied, in order, to the resulting value.

Called as:

date.start()
date.start(zone)
date.start(zone, offsets...)

Parameters:

zone
Type : Text
Default Value : UTC
Description : A time zone ID (as defined by ZoneId) to which the timestamp is converted
offsets...
Type : Text
Default Value : No offsets
Description : Optional offsets to apply to the timestamp, applied sequentially

Return type:

Date

Returns:

The context creation timestamp adjusted to the specified zone and offsets

Throws:

DateTimeException If the zone is not a valid time zone ID

See:

Since:

1.0
